# CARMIX Next-Era Roadmap: Six Dependency-Ordered Cycles Toward a Usable General-Purpose OS

Honest, tier-labeled. Every claim is marked one of three ways.

- PROVEN: green in the repo today (single-CPU/QEMU), backed by a test or a machine-checked proof.
- IN-PRINCIPLE: the primitives it composes are proven, but the composition itself is not built yet.
- OPEN: not designed or not solved, or hardware/effort-blocked.

Prior art is credited throughout. CARMIX's genuine contribution is never the isolation itself. It is the unification of already-known ideas onto one content-addressed store plus one machine-checked anti-amplification gate, with proof reuse across layers.

## The killer whole-system capability (target, IN-PRINCIPLE)

The System Root: one 32-byte BLAKE3 hash that names the entire live software system at a quiescent epoch boundary. Fold the roots CARMIX already maintains (each process's page-table-inclusive task-hash, each MVCC capability-slot version, the content-store manifest, the content-addressed FS root) into one Merkle-DAG root, recomputed diff-proportionally at each epoch boundary through the existing epoch tree.

Once the running software system is a single value, two capabilities that no POSIX kernel offers become possible, each reusing machinery already built and measured:

1. Whole-system time-travel and branch. Undo or fork the entire running software system, processes included and not just files, by pinning and rematerializing a past root. Storage per checkpoint is diff-proportional because consecutive roots share unchanged subtrees.
2. Whole-system attestation. Sign the root with the existing Ed25519 path so a remote verifier checks one signature and re-derives one root to learn exactly what code is running and the precise authority ceiling every component holds.

Why this is hard on Linux. Live kernel state is scattered mutable pointer-linked objects with no content identity, so CRIU and VM snapshots are opaque full images with no substructure to diff. Filesystem snapshots (ZFS, btrfs, Time Machine) do dedup across snapshots via copy-on-write shared blocks, but they cover files only and never running process state. The genuine CARMIX delta is content-identity of live process state, not dedup of files. TPM, DRTM, and IMA measure code that was loaded. Confidential-computing attestation (SEV-SNP, TDX) and control-flow attestation narrow the runtime gap. The precise CARMIX novelty is per-component capability-ceiling binding folded into the same hashed manifest, so the attested value states not only what code runs but what authority each component currently holds.

Status honesty. This capstone is IN-PRINCIPLE. The lower-cycle primitives it composes are PROVEN, but the System Root object, the retention policy, the rematerialize-at-root entry point, and sign-or-verify-the-root are unwritten. Nothing "falls out for free" until that composition is built.

Uniform honest boundary, stated once up front. The root would name the SOFTWARE object graph at an epoch boundary. It would not name device, DMA, firmware, or microcode state, nor the instants between epochs. Time-travel obeys the output-commit problem (Elnozahy et al.): it cannot un-send bytes that already left the NIC or un-write an external device. The attestation root of trust is EMULATED (software Ed25519 standing in for a TPM). Real hardware-anchored trust plus a trustworthy monotonic clock for freshness are OPEN, gated on physical bring-up.

## The one measured wall that gates the whole stack (PROVEN)

Activate or rematerialize-by-hash is 2 to 3 orders of magnitude costlier than the direct fast path. So CARMIX content-addresses ONLY at coarse boundaries (a pipeline, a window subtree, an app closure, an epoch) and keeps every interactive path (cursor, keystroke, caret, animation) on the direct blit or switch. Re-rooting a depth-2 tree per write costs roughly 71k cycles, which is why the mutable DATA plane is deliberately kept off per-write re-rooting.

## The uniform honest ceiling running through all six cycles

Content-addressing buys identity, integrity, atomic update, free rollback, restart, and provable authority-bounding for the IMMUTABLE code and authority plane. It buys nothing for:

- the mutable DATA plane. User files, config, and logs do not want per-write re-rooting or code-style rollback. Separating a rollback-able code root from a forward-only data store is OPEN.
- hardware and driver breadth. GPU firmware blobs, xHCI/USB-HID, Intel HDA, and a real NIC are irreducibly large and hardware-blocked. The model bounds and restarts drivers. It does not shrink them. Only PS/2 plus emulated NVMe and VT-d exist today.
- liveness. A resident or materialized thing can hang. Watchdogs are non-content-addressed live state.
- real energy. Needs RAPL/ACPI counters QEMU does not model. Any joule figure is a stated proxy.
- covert channels. The dedup timing channel is named, not solved.
- output-commit. Time-travel cannot un-send.

"No ambient root" is PROVEN for the authority ALGEBRA (the order has no top element, the only mint moves strictly down). It does not mean "no privileged party". It relocates the trusted base to explicit out-of-band key custody plus physical control of the raw store. That base is smaller than Unix's always-live uid 0, but it is not nothing.

Scope of evidence. DRCC and convergence results are interim unreviewed preprint theory. All demos are single-CPU/QEMU.

## Cycle 1: Capability identity and least-privilege authority (the backbone)

Thesis. A user or tenant is a capability DOMAIN, not an /etc/passwd row. A domain is a U-4 delegation root keyed to an Ed25519 identity in the trust store. Login is capability acquisition: a challenge-response proof of key possession that yields the domain's root swcap (a half-open range plus permission set plus validity the gate already understands). Groups, roles, and ACLs are delegation SUBTREES. Sub-delegation is strictly attenuating (A(256) to B(128) to C(64)). Revocation walks the recorded chain transitively. An application's requested permissions ARE the delegated capability subset it is spawned with, and requesting more than delegated is impossible-by-construction under the machine-checked anti-amplification bound.

Why CARMIX-native and why it matters. Unix authenticates an identity then consults ambient tables (uid/gid, DAC, capabilities(7)) that uid 0 bypasses, and setuid moves UP the lattice. CARMIX login acquires authority directly and the only mint moves strictly DOWN, so "root" has no denotation because the order has no top element to occupy or bypass. This is the KeyKOS, EROS, seL4, and Miller POLA lineage. The native parts are the machine-checked anti-amplification gate and content-addressed image identity. The "algebraic no-top order" is a restatement of the ocap no-ambient-authority principle already realized in EROS and seL4, so it is a thin standalone novelty. The real contribution is wiring it into how identities log in and apps spawn, with the bound checked in proofs/Carmix.v.

Build path (reusing existing primitives). Foundation only, no proven-core edits. Reuses: the content store, the anti-amplification swcap gate and its Coq proof, U-4 delegation chains and chain-walking revocation, spawn-from-content-addressed-image, capability-gated FS open, vendored tweetnacl Ed25519 plus the trust store, and the distrusting-migration quote/verify path for attested sessions.

Tier: PROVEN (single-CPU/QEMU). Two identities log in to two disjoint capability roots. A process under A requesting a cap into B's range is refused (dest authority not <= source). An over-broad re-mint past a ceiling is refused with the migration status codes. A forged Ed25519 login fails verification.

Dependencies. None. This is the base layer.

Honest boundary. "No ambient root" holds for the authority ALGEBRA, not the whole system. The first identity key is baked in, and who provisions it out-of-band is the one irreducible bootstrap trusted base. The machine owner controls the raw store and, with no persistent anti-rollback state (memory-only today), can wipe or roll back domains, so that owner is a de-facto trusted base. The gate bounds over-reach, never the initial grant decision. An over-broad first grant or a confused deputy is fully within the model, and capability UX is historically where users fail. Real friction versus POSIX: "list all members of group G" is a delegation-forest walk, and cross-app cooperation needs explicit capability passing that ambient authority gives for free. Attested-session integrity is EMULATED. Real login integrity needs a hardware root of trust and a trustworthy clock, both OPEN.

## Cycle 2: Content-addressed software distribution

Thesis. An application is the root hash of its dependency-closure Merkle DAG. Every edge names an exact BLAKE3 hash, so the closure is fully resolved by construction. Install is rematerialize the closure by fetching only objects the destination lacks (the diff-proportional migration transfer), re-verified by re-hash on read. Two library versions are two coexisting hashes, so a diamond dependency is no conflict and there is no SAT/PubGrub solver and no lockfile-versus-manifest drift. The whole installed system is a Merkle directory mapping app-names to app-root-hashes. An update publishes new content and re-roots that directory while the old root still resolves the entire old system, and the switch is a single durable-root update ordered after content lands. Rollback is re-pointing the durable root at a retained system hash, sharing every unchanged object via dedup.

Why CARMIX-native and why it matters. This is Nix, Guix, OSTree, and OCI functional deployment, but the store, verify-on-read, dedup, and GC are PROVEN kernel primitives rather than a userland daemon over a mutable ext4 plus SQLite it must trust. The native delta is only that store plus verify plus GC are kernel primitives. dpkg, apt, and rpm exist to arbitrate ownership and version conflicts inside one mutable global /usr. Content-addressing deletes the namespace, so it deletes the conflict and the solver. Rollback is the absence of destruction, not the reversal of it.

Build path (reusing existing primitives). Reuses: content store plus BLAKE3 dedup, U-4 Merkle namespace (directory = name to hash), the migration diff-proportional transfer path, domain-partitioned dedup, durable refcount GC, FS_LOG re-root, and persistence (durable root survives cold reboot).

Tier: PROVEN (single-CPU/QEMU). Represent a multi-object app as a Merkle-DAG root. "Install" by transferring only missing blocks between two store states. Store two library versions and show both hashes coexist. Upgrade by re-rooting the installed-set and show the old root still resolves the entire old set. Crash between content-write and root-update leaves the old root intact. Rollback is re-point, O(1). Drop an app root and watch GC reclaim exactly the now-unreferenced blocks while shared deps survive.

Dependencies. Cycle 1 (install authority is a delegated attenuated capability subset).

Honest boundary. Content-addressing gives reproducible DISTRIBUTION, not reproducible BUILDS. The source-to-binary step must itself be deterministic, which is the hard-won part of Nix and Guix and is not free here. Coexisting hashes eliminate NAME conflicts, not ABI or semantic conflicts. "Which hash should I move to" needs a mutable trust-rooted name-to-hash index, which is exactly where supply-chain trust re-concentrates and is OPEN. Poor fit for the mutable DATA plane. Real network fetch of a remote closure is OPEN (transport is ivshmem/local today).

## Cycle 3: Rematerialized system services, power, and bounded drivers

Thesis. Not three new subsystems but two proven primitives (rematerialization/residency and the anti-amp gate) aimed at new targets. INIT is a materializer, not a trusted PID-1 daemon. A service is the hash of (code plus config plus capability-set), and init hands it a capability re-minted from init's own ceiling, so launching a service with more authority than init holds is structurally impossible. The dependency graph IS the acyclic GC-refcounted content graph. Restart-on-crash is rematerialize from the last durable checkpoint. Config change is a new hash. POWER: dematerializing a process is a suspend, so hibernate is dematerialize the live set into the store and power off, and resume is rematerialize by hash through the demand-paging fault handler (BLAKE3-verified, fail-closed). The hibernate image is a dedup'd per-block-verified Merkle DAG, not an opaque blob. DRIVERS: the IOMMU is the hardware realization of the anti-amp gate, where a driver's DMA window is a swcap minted through the proven gate and the IOMMU page table is programmed to exactly that window, so a bus-mastering device physically cannot DMA outside its driver's capability.

Why CARMIX-native and why it matters. One rematerialization operation is service-restart, process-suspend, and system-hibernate at once, and one anti-amplification theorem holds from the software swcap check down to the IOMMU page table. systemd is an ambient-root PID 1 with mutable named units. Here the unit IS its hash. Linux hibernate (swsusp) is an opaque wholesale writer with no dedup. CARMIX hibernate is the ordinary content-store path, so it gets per-block verify, cross-process dedup, and free incremental hibernate. Linux IOMMU use is an optional bolt-on. Here the DMA firewall is the same "destination authority <= source" property already proven. Prior art: Nix generations, Fuchsia capability-routed components, MINIX 3 restartable drivers, Erlang supervision, seL4, CRIU/swsusp. The native contribution is only the fusion under one remat op plus one ceiling.

Build path (reusing existing primitives). Reuses: content store, swcap gate, residency manager plus durable checkpoint, demand-paging fault handler, heap page-dedup plus COW, acyclic refcount GC plus tombstone, MVCC gate as config staging/commit, ACPI parse (MADT built, extends to DMAR), and the emulated-NVMe/VT-d QEMU harness.

Tier: PROVEN (single-CPU/QEMU) for the software path. A run_init() stage materializes N services by hash, refuses a service requesting authority wider than init's ceiling, crashes one and rematerializes from checkpoint, switches a config-hash then rolls back, and GC-reclaims a dead dependency subtree. Whole-live-set dematerialize-to-store plus rematerialize-by-hash-with-verify (hibernate) with cross-process page dedup and tamper-fail-closed on resume. Per kernel/TASKSTATE_LOG.md (TS0) the page-table hierarchy is now dematerialized and its frames freed.
Tier: IN-PRINCIPLE for the emulated-VT-d rung (parse DMAR, program an IOMMU domain to a swcap-minted DMA window, refuse DMA outside it). It is emulated VT-d, not physical.

Dependencies. Cycle 1 (init re-mints service caps from a ceiling, drivers hold delegated DMA caps) and Cycle 2 (services are content-addressed, config-as-hash).

Honest boundary. Identity, integrity, and restart come free. Liveness does not: a service can be resident-and-hung, and detecting that needs a heartbeat or watchdog, which is genuinely non-content-addressed live state (OPEN). Config-as-hash means every knob edit mints a new identity, which is real UX friction versus systemctl set-property. Dataflow ordering cannot express a pure timing dependency, so timer-units stay conventional. The residency break-even is computed live today, but it is NOT a C-state or idle governor until its cost term carries joules, and QEMU does not model RAPL/ACPI, so any joule figure is a proxy. That energy-aware governor is OPEN. Content-addressing gives zero shortcut on driver SIZE. GPU, USB (xHCI/hub/HID), audio (HDA), and a real NIC are irreducibly large and hardware or effort-blocked (OPEN). Driver restart restores software state, not device hardware state.

## Cycle 4: CARSH, the authority-routed memoizing shell

Thesis. The shell becomes a constructor of nodes in the Merkle-DAG-of-live-execution the kernel already maintains. A command is a content hash, a pipeline is a DAG of content-addressed stages, and the pipeline node is the hash of (ordered stage image-hashes plus input object-hashes plus args). Three consequences. (1) Memoization: re-running a pipeline with unchanged command and input hashes is a store get, and a command's declared inputs ARE its capability set, so an undeclared read is structurally impossible and input-completeness is a kernel invariant, not a sandbox promise. (2) Least-privilege composition: a pipe is a capability IPC channel and each stage receives only the attenuated caps the shell delegates, so an accidental "rm -rf my home" is impossible if no write cap to home was delegated. (3) Time-travelable automation: the session is itself a content-addressed live execution, so "undo the last command" rematerializes prior session state by hash rather than by replay, and a cron job becomes "at T, rematerialize pinned state H and apply command C".

Why CARMIX-native and why it matters. Nix, Guix, Bazel, Buck2, Shake, and redo memoize over impure ambient-authority processes whose soundness only approximates input-completeness by sandbox convention, so a loose sandbox silently poisons the cache. Unison content-addresses pure code definitions. CARMIX extends content-addressing to whole LIVE EXECUTIONS with capability-bounded side effects. Object-capability shells (Shill, Plash, Capsicum) carve down ambient authority on a substrate that grants it by default. CARMIX has no ambient authority to carve. Record-replay time-travel (rr, Pernosco, UndoDB, WinDbg TTD) reaches a past state by re-executing under a non-determinism log. CARMIX reaches it by addressing its hash. The genuine native delta (capability set equals declared inputs, so input-completeness is a kernel invariant) is real but narrow.

Build path (reusing existing primitives). Reuses: spawn-from-image, store put/get plus BLAKE3, capability IPC (data plus cap cross, cross-core), domain-partitioned dedup for memo storage, the Merkle namespace (cwd/PATH/history as directory objects), rematerialization/migration fetch plus verify plus re-mint, GC roots for pinning history, DRCC for cross-run reproducibility (interim).

Tier: PROVEN (single-CPU/QEMU) for the core spine. A 3-stage pipeline of spawn-from-image stages wired by capability IPC with content-addressed outputs. A second run served by store get (memo hit as hash equality). A DRF pipeline reproducing the same output hash across runs. A stage that faults on over-reach (granted store-read works, un-granted store-write faults across INT 0x80). A revoked shell grant reaching a descendant stage. Pin a session state as a GC root and rematerialize it by hash.
Tier: IN-PRINCIPLE for the whole-session time-travel tooling (DB1-DB6: recorder and differ not built).
Tier: OPEN for the parser/REPL frontend, TTY line-editing and job-control, memo retention/GC policy, and any stage with clock/net/device caps (unsound to memo).

Dependencies. Cycle 1 (per-stage attenuated authority, revocation via U-4-3), Cycle 2 (commands are content-addressed), Cycle 3 (stages are materialized services).

Honest boundary. Memoization and reproducibility are sound only for the deterministic, data-race-free, capability-closed spine. A stage granted clock/RNG/net/device is not a pure function of its inputs and must be excluded from the cache. The capability set flags which stages are impure, which is a native advantage, but it does not make them memoizable. The cost crossover bites hardest here: this wins at coarse grain (an expensive build stage, a whole pipeline), never per-keystroke. The memo cache is un-reclaimable while retained and the retention policy is OPEN. POLA usability is a documented cost. The gate bounds explicit authority, not covert channels. Time-travel obeys output-commit: it recovers internal state and cannot un-send a packet.

## Cycle 5: The capability desktop

Thesis. The human-interaction layer with no ambient screen or input authority and no new GUI security kernel. SPATIAL authority: a window or surface, a framebuffer rectangle, and an input-focus grant are all capabilities minted through the same anti-amp gate that bounds migration and syscalls, so the compositor becomes a thin multiplexer over store objects and paint/read of a rectangle can only be attenuated down a chain, never amplified. TEMPORAL authority: the whole scene (window tree, z-order, focus, per-window subtrees, tiled framebuffer) is an immutable content-addressed DAG and each frame is one BLAKE3 root, so unchanged subtrees share by hash and recompositing is diff-proportional. Four jobs collapse into rematerialize-by-hash: damage-tracking (diff two roots), undo/session-scrub (rematerialize a prior root), compositor crash-recovery (remat the last root bit-identically), and live desktop-session migration (send only tiles the destination lacks). DYNAMIC authority: every input event is a content-addressed object delivered as a re-mint bound to the target surface's capability, so synthetic-injection and cross-window eavesdropping are refused by construction, and a trusted path is simply the input capability whose chain terminates at the trusted compositor root.

Why CARMIX-native and why it matters. X11 has ambient screen and input authority, so keyloggers are trivial. Wayland removes that but re-implements the entire security policy inside a bespoke compositor, a new trusted base with ad hoc rules and no external proof. The CARMIX delta is economy: window authority IS the anti-amplification relation proven once, so the compositor's trusted base shrinks to a multiplexer and the same theorem covering migration, faults, and syscalls covers pixels. Functional UI (Elm, React vDOM, ImGui) treats the view as a pure function of state and diffs it, and content-addressed DAGs with structural sharing are old (Git, Nix, Okasaki). The native part is content-addressed rematerializable and migratable LIVE UI state tied to a proven authority re-mint. The no-ambient-authority goal itself is ALREADY achieved by EROS-TWS, Genode nitpicker, and Qubes, so that is not the contribution.

Build path (reusing existing primitives). Reuses: the swcap gate plus U-4 chains, the existing software desktop (window_t, draw_window, compositor, PS/2 input), content store, remat (test R3), diff-proportional migration (net_repro), durable GC for history pruning.

Tier: IN-PRINCIPLE (single-CPU/QEMU). window_t becomes a gate-minted capability, sub-rectangle delegation succeeds and over-grant is refused with migration status codes. A tiled scene with per-frame root hash shows diff-proportional repaint on drag, scrub-to-prior-frame remat, and compositor kill/restore from a root. PS/2 events wrapped as hashed capability-stamped objects delivered only to the focused holder, with non-holder delivery and forged over-authority events refused, plus a store-backed provenance log. (Primitives are green, the desktop wiring on top of them is the unbuilt part.)
Tier: OPEN and hardware-blocked for GPU/KMS, hardware cursor plane, dma-buf, USB-HID, multitouch, and text/font beyond the 8x8 bitmap font.

Dependencies. Cycle 1 (surfaces and input are delegated capabilities) and Cycle 3 (GPU/USB-HID drivers, the hardware-blocked part). Its per-frame scene roots feed the Cycle 6 System Root.

Honest boundary. The Linux-Mint usability bar is an ECOSYSTEM and RENDERING problem (GPU/KMS, real fonts, USB-HID, toolkits, a browser, apps), not an interaction-model problem. Content-addressing conjures no drivers and no apps. Capabilities bound WHO may paint or read a rectangle but not visual SPOOFING inside a legitimately granted rectangle. Defeating that needs trusted labeling or colored borders (nitpicker, Qubes), which capabilities alone do not provide. Provenance stops injection and eavesdropping, not perception. The cost crossover is a hard wall: hashing a full framebuffer at 60Hz is infeasible and per-event hashing is a real cost for high-rate input, so content-addressing lives only at coarse boundaries and the interactive path must stay on the direct-blit fast path. "Infinite undo" is really undo bounded by GC retention.

## Cycle 6: The System Root (the capstone, IN-PRINCIPLE)

Thesis. Compose the roots every lower cycle already emits into ONE system-wide Merkle-DAG root: a manifest of {each live task's page-table-inclusive task-hash, each MVCC capability-slot version, the store manifest, the FS root}, computed diff-proportionally with the existing epoch tree so only subtrees whose children changed re-hash. At each quiescent epoch boundary the running software system is one 32-byte name. From that, two whole-system capabilities would compose. (1) Time-travel by hash: retain a bounded history of durable roots pinned against GC, then rematerialize the entire software system (or one process, or the FS) at a past root, with diff-proportional marginal cost because consecutive roots share subtrees, and branch by pinning a root and starting a fresh epoch chain from it. (2) Attest the root: sign the system root with the existing Ed25519 authority key so a remote verifier checks one signature and re-derives one root to learn exactly what code and what authority (capability ceiling, inside the hashed manifest) every component holds. Flipping one ceiling changes a task-hash, changes the root, and the signature no longer verifies.

Why CARMIX-native and why it matters. No conventional kernel has a verifiable content identity for its whole running state. Linux live state is mutable pointer-linked scattered structures, so a VM or CRIU snapshot is an opaque full blob with no substructure identity, and two states cannot be diffed, deduplicated, or shown ancestor/descendant by hash. Merkle DAGs and git give this for STATIC data. CARMIX gives it for LIVE execution because it already hashes the page-table-inclusive process. "Undo the last N minutes of my whole computer, including running programs" is out of reach on Linux: CRIU and VM checkpoints are full and opaque, filesystem snapshots cover files only, and Nix generations roll back installed packages, not running state. TPM/DRTM/IMA measure code loaded, and confidential-computing attestation narrows the runtime gap, but the precise CARMIX novelty is per-component capability-ceiling binding folded into the hashed manifest.

Build path (reusing existing primitives). Composes the roots lower cycles emit: Cycle 1 ceilings (inside the manifest), Cycle 2 system/FS root, Cycle 3 service checkpoints, Cycle 4 session states, Cycle 5 per-frame scene roots. Reuses: store/epoch_tree.c (incremental equals full, proven), the per-task page-table-inclusive task-hash, MVCC version words, persistence ROOT-block format, durable GC refcounts/tombstone/resurrection window to pin retained roots, rematerialization/demand-paging with BLAKE3 verify, the two-QEMU ivshmem migration harness, and vendored tweetnacl Ed25519 plus pre-shared keys. New code is only a top-level manifest object, a retention policy, a rematerialize-at-root entry point, and sign/verify-the-root. No new crypto, no new store, no proven-core edit.

Tier: IN-PRINCIPLE. The primitives above are PROVEN. The four new pieces are unwritten, so the capstone is not demonstrable today. Once built, the intended demonstrations are: build a system root over 2 live processes plus store manifest plus FS root, mutate one page, recommit and show only the touched leaf-to-root path re-hashed. Rematerialize a process at a retained past root and show its re-derived root matches, with diff-proportional history storage. Sign a root at A, verify at B, then flip one capability ceiling and show verification reject.

Dependencies. All lower cycles.

Honest boundary. The root names the SOFTWARE object graph AT AN EPOCH BOUNDARY, not the whole machine and not the instants between epochs. Three structural limits. (1) It requires mutators quiescent at the boundary, a stop-the-world barrier on one CPU. On SMP this is a global consistent-cut problem, precisely the DRCC research line, which is UNVALIDATED until SMP is used (OPEN). (2) Device, DMA, LAPIC-timer, and in-flight-DMA state is not content-addressed and is OUTSIDE the root, so "whole machine" would overclaim. (3) Cadence trades hashing cost against staleness. Time-travel is sound only for in-system content-addressed state and obeys the output-commit problem: it cannot undo bytes already on the NIC or a write to an external device. Honest framing is "undo all in-system state up to the last externalized output". Retention is a space and policy tradeoff (a window, not unbounded history). Rematerializing a process that held a live NVMe queue does not restore the controller. Attestation binds the software object graph, NOT the CPU, microcode, or firmware executing it. The root of trust is EMULATED (software Ed25519), so a hardware TPM/DRTM is still needed to attest that CARMIX itself and the real BLAKE3/Ed25519 are running, and the baked first-authority-key bootstrap remains the one irreducible assumption. It is an epoch-boundary snapshot, so a TOCTOU window exists between the attested root and the next mutation. Attestation is per-epoch-boundary, not continuous. Truly continuous attestation needs the unbuilt SMP consistent-cut (OPEN). Freshness needs a nonce plus a trustworthy monotonic clock, both OPEN (the kernel has only a tick counter), so replay of a stale-but-valid root is not yet closed.
