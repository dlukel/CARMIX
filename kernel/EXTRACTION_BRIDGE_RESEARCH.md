# EXTRACTION BRIDGE RESEARCH

Attacking the one caveat on every honest CARMIX claim: the Coq proof in proofs/Carmix.v is about the ABSTRACT re_mint model, while the running gate in carmix/swcap.c (cvsasx_swcap_check and cvsasx_sw_cap_remint) is C tested against attack tables. This packet asks how far that gap closes if the PROVEN re_mint is extracted into a reference oracle and the C is differentially checked against the exact function the proof quantifies over, and it ranks that honestly.

Discipline in force: READ-ONLY grounding, no repo edits, no source change to the byte-frozen proofs/Carmix.v (D4). Agreement is evidence, never refinement (D3). The reference must come FROM the extractor, not a hand retype (D7). seL4-scale full functional C refinement is the fenced gold standard and is not the target (D2/D5). This is a verdict exercise, not a survey (D5). Promotional framing is banned, deltas only (D1).

## X1 EXTRACTION FEASIBILITY TEST

Question: can re_mint actually be extracted from the frozen proof into an axiom-free runnable OCaml term without editing proofs/Carmix.v, and is that term the exact object the theorem is about.

### Findings

The entire dependency cone of re_mint (proofs/Carmix.v:129-133) is computational and Prop-free. The helpers are all Definition or Fixpoint over bool, nat, and list: Nat.max and Nat.min from stdlib, Pinter (:81-82, filter over Pmemb), Pmemb (:72, existsb over Perm_eqb), Perm_eqb (:38-48), has_wx (:116), has_forbidden (:119), reject (:122), is_forbidden (:55-123), the Cap record (:94-99), and the seven-constructor Perm inductive (:29-36). The Prop layer (cap_le :105, Psubset :69, Pmem :67) and Section ObjectGraph with Variable Identity and Hypothesis Identity_injective (:298-301) are NOT in the term and are erased. Lia and Arith are imported but inject no axioms into computational terms, since the lia proofs are Prop and are erased. So Extraction re_mint is axiom-free and IS the exact object anti_amplification (:139) quantifies over.

Extraction is driven from a NEW file (for example a proofs/ExtractRemint.v that does From Carm Require Import Carmix. Extraction "remint.ml" re_mint.). The Makefile already maps -R HERE Carm, so Carmix is Requirable as Carm.Carmix. proofs/Carmix.v is not touched, so D4 is satisfied and no deferred future directive is needed for the faithful build.

One faithful-build property to state up front: default extraction keeps Peano nat as unary O and S, so 2^64 successors are infeasible to construct and the oracle is only runnable at small magnitudes. Making it run at machine width needs ExtrOcamlNatInt or a bignum remap, which is no longer the proven function at the overflow boundary. That is a real ceiling, deferred, not crossed.

### Verdict

EXTRACTABLE-AS-IS. re_mint extracts verbatim to an axiom-free OCaml term from a separate driver file with the frozen source untouched. This is the load-bearing positive result of the whole packet: D7 is genuinely satisfied, the reference comes from the extractor.

### Honest boundary

The extracted term is the proven function only at unary-nat widths. Extraction succeeding does not by itself check anything about the C, it only produces the oracle the later fronts consume. The extractor plus the OCaml compiler and runtime are in the trusted base and are not verified here (CakeML would close this and is not used).

## X2 THE REFINEMENT QUESTION

Question: for the extracted re_mint against carmix/swcap.c, rank three tiers, (a) differential agreement, (b) hand algorithmic-equality proof, (c) machine-checked refinement.

### Findings

Scope. Only one C function has a proven executable counterpart. re_mint corresponds to cvsasx_sw_cap_remint (swcap.c:56), both produce a cap from source plus request. cvsasx_swcap_check (swcap.c:98, the enforcement gate the CBMC rung targets) corresponds to cap_le, which is a Prop (uses Psubset, a forall), so it is not executable and there is nothing to extract for the check gate. The extraction-differential idea applies to the MINT path only. What extraction adds over today: cvsasx_sw_cap_remint currently has NO reference-checked coverage at all. The CBMC harness in proofs/cbmc/swcap_check_harness.c calls only cvsasx_swcap_check (line 67) against a hand-written inline spec (lines 78-81), and cvsasx_sw_cap_remint is compiled but never asserted against. Extraction gives the mint its first check against the exact proven function.

Three honest divergences the harness must confront.

1. CLAMP versus REJECT. Abstract re_mint CLAMPS (lo = max breq (base s), hi = min lreq (limit s)) and INTERSECTS (Pinter preq (perms s)), then accepts unless W-X or forbidden. The C REJECTS: swcap.c:71-72 return BAD_BOUNDS on out-of-range, swcap.c:78 returns AMPLIFY_PERMS if perms are not already a subset. They are different functions in general and coincide only on the already-legal slice where clamp and intersect are no-ops. On that slice out_cap.base = region.base + offset = lo, out_cap.length gives hi, out_cap.perms = pir->perms = Pinter (subset no-op). Agreement must be stated as equality on the CVSASX_OK slice, with only valid = 0 checkable on the reject slice.

2. PERM DOMAIN MISMATCH, the adapter is lossy and unproven. Abstract Perm has 7 elements. C (cap/cvsasx_pir.h:78-102) defines 13 named bits. Abstract forbidden is {Seal, Unseal, Invoke, AccessSysRegs} (4). C SW_FORBIDDEN_MASK (swcap.c:12-13) is those 4 plus SET_CID plus UNMODELLED (6), two bits the abstract model cannot even represent. Abstract W-X is Write and Execute. C W-X (swcap.c:76) is EXECUTE and WRITE_ANY where WRITE_ANY = STORE or STORE_CAP (two bits). GLOBAL, LOAD_CAP, STORE_CAP, STORE_LOCAL_CAP have no abstract image. So the adapter mapping uint32 bitmask to and from list Perm is a modeling decision that could itself be wrong and it enters the trusted base unproven.

3. WIDTH GAP. Faithful extraction keeps Peano nat. Adding ExtrOcamlNatInt reintroduces the unsound-on-overflow remapping the abstract model avoids, so honest extraction stays Peano and tests small magnitudes, which means the oracle cannot exercise the uint64 wraparound corners. Those corners are precisely what the CBMC rung covers on the real C. So extraction-differential and CBMC are COMPLEMENTARY, concrete execution against the exact proven function versus symbolic bounded decision against a hand spec, and neither subsumes the other. CBMC cannot consume the OCaml reference, so they do not compose into one artifact.

Tier (a) DIFFERENTIAL AGREEMENT, reachable now, low cost, evidence only. Feasible immediately given the extraction. The adapter must satisfy the C structural gates (version swcap.c:62, unsealed :63-64, referent-valid and hash-match :65-66, region-within-root sw_check_region :44-54) so that only the bounds and perm decisions re_mint models are exercised. The version, seal, and hash rejects are correctly outside the correspondence, since re_mint has no such notion.

Tier (b) ALGORITHMIC EQUALITY BY HAND, reachable, medium cost, low value in full. A full paper step-by-step proof that the C computes the same function as the oracle would land between the two useful rungs without either rigor: it lacks CBMC exhaustiveness, it lacks a machine-checked proof tool guarantee, no checker keeps it honest, and it rots. Its one genuinely useful sub-part is a roughly one-page correspondence table justifying the adapter of divergences 1 to 3 (why OK-slice equality holds, the exact perm-bit map, the clamp-versus-reject boundary). That memo is documentation tier (a) needs anyway. The big equality proof is not worth writing.

Tier (c) MACHINE-CHECKED REFINEMENT, not at reasonable cost, expected and acceptable. A proof that the freestanding C (or a faithful embedding) refines the oracle needs either a machine C semantics plus refinement stack (AutoCorres or seL4 style) or a separation-logic harness (RefinedC, VST, Iris in Coq). The repo has neither and no separation-logic development exists in-tree. The team has demonstrated exactly two proof classes: the Carmix.v clamp-monotonicity and finite-tree class (357 lines) and application of an off-the-shelf bounded model checker (CBMC). Interactive C-refinement is a distinct, materially heavier proof-engineering class it has not exhibited. Honest size of the target: the ceiling actually reached is the CBMC bounded decision over one roughly 15-line straight-line function (cvsasx_swcap_check). Even refining just cvsasx_sw_cap_remint (roughly 40 lines with structs, region checks, the 32-byte hash compare, the post-mint re-verify) to the oracle via RefinedC would be a specialist multi-week to multi-month effort versus CBMC stated 1 to 3 weeks. Full-system source-to-abstract refinement is the fenced seL4-scale gold standard, a whole microkernel proof rather than one gate, a distinct and much larger class.

### Verdict

Per tier. Tier (a) DIFFERENTIAL AGREEMENT is ACHIEVABLE-NOW, LOW COST, EVIDENCE-ONLY (never refinement, per D3), enabled by a genuine extraction (D7 satisfied, re_mint is Prop-free and axiom-free and extracts verbatim, D4 satisfied, driven from a new file with Carmix.v untouched). Tier (b) ALGORITHMIC EQUALITY BY HAND is REACHABLE at MEDIUM COST but NOT WORTH WRITING IN FULL, only the roughly one-page adapter-correspondence memo is worth writing as documentation tier (a) needs regardless. Tier (c) MACHINE-CHECKED REFINEMENT is NOT AT REASONABLE COST (expected and acceptable), no separation-logic harness in-tree, the team has shown only the clamp-monotonicity Coq class and off-the-shelf CBMC, not interactive C-refinement, the honest ceiling is the landed CBMC bounded decision over one gate, and full C to re_mint refinement is a separate and materially heavier class with full-system refinement being the fenced seL4-scale gold standard, a distinct and much larger class.

The H-build targets tier (a). Exact bounded statement. Let R be the OCaml function from Extraction re_mint (default Peano nat, no int-remapping axiom, extraction cone Prop-free and axiom-free, so R is the exact term anti_amplification quantifies over). Let A be a hand-written unproven adapter that (i) builds structurally-valid C inputs (cust initialized and valid, pir unsealed, version-correct, referent-hash matching region, region within root) so only the bounds and perm anti-amp logic is exercised, (ii) maps region and pir into (s, breq, lreq, preq) over the 7 modeled perms and small nat magnitudes, (iii) maps out_cap back to a Cap. Then over the finite generated set T: for every t in T where cvsasx_sw_cap_remint returns CVSASX_OK, A(out_cap) = R(A(t)) and re_mint clamp and intersect were no-ops (the C accepts only already-legal requests), and for every t where the C rejects, the only compared invariant is out_cap.valid = 0 (abstract re_mint instead clamps and intersects and may return a valid attenuated Cap, a definitional divergence not a defect). This is EVIDENCE the running mint agrees with the proven re_mint on the exercised, adapter-mapped, in-model slice. It is NOT refinement: silent on un-exercised inputs, on un-modeled bits (GLOBAL, LOAD_CAP, STORE_CAP, STORE_LOCAL_CAP, SET_CID, UNMODELLED and the STORE-or-STORE_CAP encoding of abstract Write), on the uint64 overflow corners Peano nat cannot represent, and on the correctness of A itself.

### Honest boundary

D3, agreement is not proof: every part of tier (a) is EVIDENCE the C matches re_mint on the exercised space, no number of runs upgrades it to a refinement claim, the reject-slice divergence and the unproven adapter alone forbid it. D7: satisfied, R is the extracted term so agreement is against the exact object the proof is about. D2/D5: tier (c) is not claimed, the honest target is one roughly 40-line gate function against a proven function over a finite slice, a distinct and much smaller thing than a whole verified microkernel, and even that one-function refinement is beyond the demonstrated proof class, the reachable ceiling is the already-landed CBMC bounded decision. D4: extraction needs no edit to the byte-frozen proofs/Carmix.v, a future nat-to-machine-int fidelity build would require ExtrOcamlNatInt which is itself an unsound-on-overflow trusted assumption, flagged as a separate future decision. Residual trusted base for tier (a) even at best: the Coq extractor plus OCaml compiler and runtime (CakeML would close this and is not used) and the hand-written adapter (unproven).

### Findings

re_mint input domain, precisely (proofs/Carmix.v:129, re_mint (s:Cap) (breq lreq:nat) (preq:list Perm)). Two disjoint axes.

(1) PERMISSION AXIS, finite, small, fully enumerable. Perm is a 7-element enum {Read, Write, Execute, Seal, Unseal, Invoke, AccessSysRegs} (Carmix.v:29-36). The permission-relevant inputs are perms s (a subset of the 7) and preq (a subset of the 7), the result perms are Pinter preq (perms s), and the reject decision is has_wx or has_forbidden on that intersect. Lattice size: 2^7 = 128 subsets per side, the reachable source-by-request product is 128 by 128 = 16,384 pairings, times the valid s tag = 32,768 permission-relevant input classes. That is small enough to run every point, so on this axis differential agreement is exhaustive over the whole domain the theorem quantifies over, not a sample. This is the strong claim, tethered to a concrete delta: exhaustion over the 16,384-pairing lattice versus sampling. It is honest because C1 anti_amplification, C2a valid_dest_no_wx, and C2b valid_dest_no_forbidden are universally quantified over exactly these permission inputs, so executing all 16,384 pairs against the extracted re_mint and cvsasx_sw_cap_remint covers the full quantifier range on this axis by exhaustion.

(2) NUMERIC AXIS, bounded in C, unbounded in the model. Coq base, limit, breq, lreq are nat (no overflow, infinite domain). The C carries them as uint64_t in four fields the harness varies: region->object_cap.base, region->object_length, pir->offset, pir->length (widths confirmed cvsasx_pir.h:176-177, cvsasx_cap.h:115-116/134-135). State space (2^64)^4 is roughly 2^256, not exhaustible, only bounded-exhaustive or dense sampling is feasible.

Extraction is actually available (D7 satisfied). re_mint and its entire computable cone (Pinter :81, Pmemb :72, Perm_eqb :38, has_wx :116, has_forbidden :119, reject :122, is_forbidden :55) are Gallina Definitions over nat, list, and bool, all extractable. The Props (cap_le, Psubset, Pmem) and the theorems are Prop and are erased, we do not need them, we need the function the proof is about, which extracts cleanly to OCaml. So the reference oracle comes from the extractor. D4 respected, the Extraction command lives in a separate driver file and Carmix.v is not edited. One nameable TCB item: the nat to OCaml mapping (ExtrOcamlNatInt versus a bignum) puts an arithmetic model in the trusted base, and it is exactly the seam where no-overflow-nat and wrapping uint64 diverge.

What exhaustive-lattice plus dense-bounded buys, relative to full refinement. The strongest honest statement obtainable: for all 16,384 source-perms and requested-perms pairs over the 7 modelled permissions, the extracted proven re_mint and the running cvsasx_sw_cap_remint agree on result-perms and reject-decision, that is, the C gate agrees with the proven model over the entire permission lattice by exhaustion, not sampling. Plus, on numerics, boundary-value density exposes the nat-versus-uint64 divergences the model structurally cannot express (base + offset wrap at swcap.c:82/90, the subtraction-guard thresholds at :71-72, length == 0 at :70). That is the exact class the CBMC rung targets, reached here by running the proven function instead of re-encoding it.

Clamp versus reject: the harness needs a RELATION, not equality, off the in-bounds grid. The model clamps (lo = max breq (base s), hi = min lreq (limit s)) and its validity ignores bounds entirely, a request with lreq < breq yields an inverted or empty but still valid = true cap if perms pass. The C rejects instead (length == 0 gives BAD_BOUNDS :70, offset or length past object :71-72, validity = 1 only on the full OK path including hash, version, otype, region gates). So model.valid and C.valid are different predicates that coincide only on the in-bounds sub-grid with all C gates satisfied. Consequence: field-equality is checkable only on the in-bounds numeric grid, off it the differential must check the shared anti-amplification predicate (both fail-closed, dest less-than-or-equal cap src), which both satisfy by construction. Encoding equality everywhere would produce spurious mismatches that are by-design semantic differences, not bugs.

Numeric grid shape. Boundary-value (dense at thresholds, sparse in the interior), per field roughly 7 landmark values keyed to each C branch, {0, 1, T-1, T, T+1, 2^64-1, one interior mid} where T is that branch bound (object_length for the offset check, object_length minus offset for the length check). Four fields {object_cap.base, object_length, offset, length} give roughly 7^4 = 2,401 numeric tuples. Cross with the 32,768 permission classes for a full run of roughly 7.9e7 cases (minutes for an OCaml oracle versus C), or decouple: full permission lattice by one canonical in-bounds numeric point, plus one canonical permission set by the full numeric boundary grid. The near-2^64 points are most informative: there the nat oracle (no wrap) and the uint64 C (wraps, and its post-mint verify at :90 re-adds with the same wrap so it does not self-catch) provably diverge, the differential makes the model-versus-machine gap visible rather than assumed.

### Verdict

FEASIBLE-AT-REASONABLE-COST as differential VALIDATION, delivering a strong bounded agreement statement, exhaustive over the model 16,384-pairing permission lattice and sampled (boundary-dense) over the roughly 2^256 numeric space, but constitutively NOT a refinement (D3, agreement is evidence on the exercised space, never proof that C refines the model). The exhaustive permission claim is over the model 128-subset lattice, which is a 7-bit projection of the C 2^32-value perm word, so it does not cover the C extra defined bits or the surrounding gate. Extraction of the reference from Carmix.v is available (D7 met) via a separate driver that leaves the frozen source untouched (D4 met).

### Honest boundary

Five things the exhaustive-lattice plus dense-numeric differential cannot do.

1. D3, load-bearing: even complete agreement over all 16,384 permission pairings is EVIDENCE the C matches the model on the exercised inputs, NOT a refinement theorem. No sentence may upgrade agrees on the whole lattice to refines. Exhaustiveness removes the sampling gap on one axis, it does not remove the model-to-C semantic-preservation obligation, which differential execution never discharges.

2. Permission exhaustiveness is over the model, not the C. The C perm word is uint32_t, 2^32 raw values with 13 defined bits (cvsasx_pir.h:78-102): GLOBAL, EXECUTE, LOAD, STORE, LOAD_CAP, STORE_CAP, STORE_LOCAL_CAP, SEAL, UNSEAL, INVOKE, ACCESS_SYS_REGS, SET_CID, UNMODELLED at 31. The 7-perm model is a lossy abstraction of this. Untouched by any model-driven differential: (a) the 6 extra defined bits and all 2^32 minus 2^7 raw values outside the 7-bit sub-cube, (b) the C SW_FORBIDDEN_MASK has 6 bits (swcap.c:12-13) but the model forbidden set has 4, the SET_CID and UNMODELLED forbidden arms have no model counterpart, (c) the C W-X uses WRITE_ANY = STORE or STORE_CAP (cvsasx_pir.h:123) but the model has a single Write, so the STORE_CAP arm of the W-X check is never exercised unless the encoding maps model-Write onto both C bits or runs two encodings. So exhaustive over the lattice means exhaustive over a 128-point projection, the extra C rejection machinery must be validated another way (CBMC).

3. Numeric side is not exhausted. (2^64)^4 is roughly 2^256, boundary plus dense covers branch structure only. Coq nat has no overflow, so the proven function literally cannot state the wrap behaviour, at the overflow boundary the oracle and C disagree by design, so the differential can only witness the gap, not close it.

4. Surrounding C machinery the model does not represent. re_mint model is (base, limit) plus perms plus valid, the C additionally gates on referent-hash compare hash_eq / BLAKE3 (swcap.c:66, collision-resistance, same class as Identity_injective), struct_version, otype_class, sealed, flags (swcap.c:62-65), region-versus-custodian-root sw_check_region (:44-54, 67), the post-mint VERIFY re-checks (:89-92), null-derefs, and a (base, offset, length) versus (base, limit) representation gap. Differential over re_mint model inputs says nothing about any of these.

5. seL4 guard (D2). This is a categorically weaker claim than seL4-style full functional C refinement (a whole microkernel of C to an abstract spec, machine-checked, person-years) and a distinct much smaller thing. The honest target here is: run an extracted roughly 30-line pure function against roughly 40 lines of C bounds and permission logic over a finite lattice plus boundary grid, producing evidence of agreement. It is not shaped like refinement, it is testing against the exact proven function, nothing here approaches or should be described as refinement.

### Findings

What the campaign compares. R is the extracted oracle. re_mint (Carmix.v:129-133) is fully computable Gallina, so Extraction "remint.ml" re_mint succeeds from a new file that Requires Carmix, proofs/Carmix.v stays byte-frozen, so extraction does not trip D4 (D7 satisfied, the oracle comes from the extractor). C is cvsasx_sw_cap_remint (swcap.c:56-96), the minting gate. The committed CBMC rung at proofs/cbmc/ targets the sibling cvsasx_swcap_check, not remint, and checks C against a hand-written inline spec (swcap_check_harness.c:78-81), the extracted oracle whole point is to replace that re-typed spec with the proven function on the mint path.

The three grounded sources of disagreement (verified against files).
(a) LATTICE WIDTH. Coq Perm has 7 elements (Carmix.v:29-36). C cvsasx_perm_t has 13 named bits (cvsasx_pir.h:78-102). Only 7 map: Read to LOAD (bit2), Write to STORE (bit3), Execute to EXECUTE (bit1), Seal/Unseal/Invoke/AccessSysRegs to bits 7/8/9/10. Six C bits have no model image: GLOBAL(0), LOAD_CAP(4), STORE_CAP(5), STORE_LOCAL_CAP(6), SET_CID(11), UNMODELLED(31).
(b) STRICTER C POLICY ON UNMODELLED BITS. C SW_FORBIDDEN_MASK (swcap.c:12-13) is the model forbidden {Seal, Unseal, Invoke, AccessSysRegs} plus SET_CID and UNMODELLED. C W-X (swcap.c:76) uses WRITE_ANY = STORE or STORE_CAP (cvsasx_pir.h:123), so it also rejects EXECUTE plus STORE_CAP, the model has_wx (Carmix.v:116) is Write and Execute = STORE and EXECUTE only. Both extra C rejections live on bits the 7-enum cannot represent.
(c) CLAMP versus REJECT SEMANTICS. Model re_mint clamps bounds (lo = max breq base, hi = min lreq limit) and intersects perms (Pinter), always returning a cap (validity as a tag). C rejects: out-of-object gives ERR_BAD_BOUNDS (swcap.c:70-72), non-subset perms give ERR_AMPLIFY_PERMS (swcap.c:78), zero-length gives ERR_BAD_BOUNDS, plus a whole preamble the model has no counterpart for (version swcap.c:62, sealed :63-64, referent-hash :65-66, root-containment sw_check_region :44-54). Consequence: R and C agree structurally only on the happy subspace (request already in-bounds, perms already subset of region, no forbidden or W-X bit) where clamp and intersect are a no-op, off that subspace R returns a clamped valid cap and C returns an error, by design. On the modelled 7 bits the C never fails open, its subset check pir->perms and not region->object_cap.perms (swcap.c:78) constrains all 32 bits, and post-mint verify (swcap.c:89-92) forces out_cap = request exactly, so on paper there is no modelled-bit input where C amplifies. The taxonomy value is therefore a forward regression net plus anti-spin, not a bug already visible.

The taxonomy (pre-stated) and operational discriminator. Run both sides through a fixed encoding bridge, then apply this ordered decision procedure to any disagreement.
STEP 1 (rule out CLASS 3, EXTRACTION or HARNESS ARTIFACT). Re-run the same case with every quantity below 2^10 (a value all of Coq nat, OCaml int, uint64, uint32 represent exactly). If the disagreement vanishes small and only appears at large widths, it lives at the number-encoding boundary, class 3. Signature: it moves when you change the harness (the nat-to-int extraction directive, the absolute-base to base-plus-relative-offset map, list-Perm to uint32-bitmask packing, PIR endianness, an over-tight nondet constraint), not when you change a gate input. Disposition: fix the bridge, never count as evidence about the C. The nat regime is the big one, default extraction makes nat unary (2^64 successors infeasible), so a usable oracle needs ExtrOcamlNatInt or a bignum in the harness file, which is not the proven function at the overflow boundary. So the width regime is not adjudicable by the extracted oracle at all, it is deferred to the CBMC rung, which decides it bit-precisely on the real C. This is the explicit H-to-CBMC seam: differential covers the small shared domain, CBMC covers the width and overflow regime.
STEP 2 (rule out CLASS 2, MODEL-SILENT REGION). If the disagreement survives at small values, check whether it involves (i) a permission bit outside the modelled 7, or (ii) a clamp-versus-reject or preamble-rejection difference. If yes and C fails closed (returns an error with out_cap zeroed, swcap.c:59), it is a documented boundary, not a bug: the custodian is strictly more conservative than the model and never amplifies. Examples: request carrying SET_CID or UNMODELLED (C rejects, model cannot see it), EXECUTE plus STORE_CAP (C W-X rejects, model silent), any out-of-bounds or non-subset request (C errors, model clamps to a valid narrower cap). Disposition: record as a boundary line, do not fix the C.
STEP 3 (CLASS 1, GENUINE C BUG, reported before anything else). What remains: a disagreement at small values, on the modelled 7 bits, where C returned CVSASX_OK with an out_cap that violates cap_le versus region->object_cap, that is out_cap.perms not subset of region perms over the modelled bits, or [out_cap.base, out_cap.base + length) not within region range, or out_cap holds a modelled-forbidden bit or a modelled W-X pair while valid = 1, equivalently on the agreement subspace C OK output differs from R valid output on a theorem-constrained field (base, limit, perms, valid) in the amplifying direction. This is the exact negation of anti_amplification (Carmix.v:139) instantiated on the real types. Report first, fix.

The ordering is the safety net. Classes must be ruled out in order 3 then 2 then 1: you may call something a class-1 bug only after the harness (3) and model-silent (2) filters fail, otherwise you cry wolf, and conversely any residual that lands on the modelled cap_le predicate is reported as class 1 before any class-2 or class-3 bookkeeping. Comparison is on the safety predicate cap_le (over the 7 modelled bits), plus structural field-equality restricted to the agreement subspace and to cases where both sides are valid (R does not zero base and limit on invalid, C does, comparing those fields when invalid is itself a class-3 trap).

### Verdict

PROVABLE-UNDER-NAMED-ASSUMPTIONS. The three-class taxonomy is a decidable classifier over R-versus-C disagreements given a fixed bridge, and it guarantees class-1 (amplification) bugs surface before anything else. Named assumptions: (1) encoding-bridge faithfulness, the offset/base, list-Perm to bitmask, and valid/status maps are reviewed and not over-constrained (an over-tight harness manufactures false agreement, D3). (2) frozen-before-campaign spec, the 7-bit modelled projection and the enumerated fail-closed regions are pinned before any run, without this a real class-1 bug can be relabelled class-2 model-silent after the fact, which is precisely the failure this front exists to block. (3) width deferral, the extracted oracle can only be run either as the proven unary-nat function (tiny inputs) or as an ExtrOcamlNatInt remap that is no longer the proven function, therefore the near-overflow regime is not covered by the differential and is handed to the CBMC rung. (4) D3 standing, agreement on the exercised subspace is evidence the C matches the proven model there, never a refinement proof. Under these four the classifier is constructible and complete over the shared small-value domain, it is not provable outright because assumptions (1) to (3) are unremovable engineering and semantic seams, and not not-at-reasonable-cost because extraction of re_mint plus a comparison harness is a bounded, days-scale artifact.

### Honest boundary

seL4 guard (D2): this is differential testing plus a disagreement classifier over one roughly 40-line function shared safety predicate, cap_le (Carmix.v:105-108) projected onto 7 permission bits over the agreement subspace, NOT functional refinement of swcap.c against Carmix.v. Full seL4-scale C-to-abstract functional refinement of the gate is not at reasonable cost for this team and is not attempted, nothing here is shaped like it. The differential proves nothing (it is testing, D3), it produces (i) bounded evidence of R-C agreement on the small-value happy subspace and (ii) a pre-committed net that routes every disagreement to bug, boundary, or artifact. Two irreducible ceilings, both named not crossed: (A) the extracted oracle is the proven function only at unary-nat widths, at machine widths it is an ExtrOcamlNatInt remap the proof does not cover, so the width and overflow regime (including the unguarded root.base plus root.length wrap latent behind swcap.c:82) is deferred to CBMC, not to this campaign, (B) even perfect agreement is only over the exercised inputs and only on the 7 modelled bits, the 6 unmodelled C bits and the clamp-versus-reject semantics are permanently model-silent, closed by C design, never by the proof. On the current C the classifier will in practice emit only class-2 and class-3 dispositions (no modelled-bit input makes the fail-closed gate amplify), its class-1 lane is a regression tripwire for future edits, and its real deliverable today is preventing a future class-2 or class-3 disagreement from being spun either as the C is broken or as agreement proves the C.

## X5 THE UNIFIED CLAIM

### What machine-checked may honestly BECOME if the bridge lands

Today the honest phrasing is narrow: proofs/Carmix.v proves properties of the ABSTRACT re_mint model, and carmix/swcap.c is C tested against attack tables plus one CBMC rung that checks cvsasx_swcap_check against a hand-written re-typed spec. The mint path cvsasx_sw_cap_remint has zero reference-checked coverage.

If the extraction bridge lands, machine-checked may honestly become this and no more: the running mint gate cvsasx_sw_cap_remint AGREES WITH THE EXTRACTED PROVEN MODEL re_mint, where the reference is the exact term extracted from proofs/Carmix.v (axiom-free, Prop-free cone, so it is literally the object anti_amplification quantifies over, not a re-typed copy). The agreement is EXHAUSTIVE over the full 128-by-128 permission lattice (all 16,384 source-by-request pairings over the 7 modelled permissions, by enumeration not sampling) and DENSE over boundary-value numeric ranges (roughly 7 landmark values per field near each C branch threshold). It is a DIFFERENTIAL result. It is EVIDENCE the C matches the proven model on the exercised, adapter-mapped, in-model slice. It is not a refinement, not verification of the C, and not seL4-scale.

The deliverable is a bounded, days-scale artifact: (i) a separate extraction driver leaving Carmix.v byte-frozen, (ii) an OCaml or C differential harness with a reviewed but unproven adapter, (iii) the roughly one-page adapter-correspondence memo justifying the perm-bit map and the OK-slice clamp-versus-reject boundary, (iv) the X4 pre-committed divergence classifier that routes every mismatch to bug, boundary, or artifact in the fixed order 3 then 2 then 1. In practice on the current fail-closed C the classifier emits only class-2 and class-3 dispositions, so its class-1 lane is a forward regression tripwire, and its present value is anti-spin.

### KILL-LIST, what still cannot be claimed

1. NOT seL4-scale refinement. Above all, no claim of full functional C refinement of swcap.c against Carmix.v, and nothing shaped like it. Even single-function refinement of cvsasx_sw_cap_remint is a distinct, materially heavier proof class the team has not exhibited.
2. NOT verification of the C. Differential agreement is evidence on the exercised space, never a proof that the C refines or satisfies the model (D3). No number of runs upgrades it.
3. NOT coverage of the check gate. cvsasx_swcap_check corresponds to cap_le, a Prop, not extractable, so the extraction-differential touches the MINT path only. The check gate reference stays the hand-written CBMC spec.
4. NOT the full C permission space. Exhaustion is over the model 128-subset lattice, a 7-bit projection of the C 2^32 perm word. The 6 unmodelled bits (GLOBAL, LOAD_CAP, STORE_CAP, STORE_LOCAL_CAP, SET_CID, UNMODELLED), the extra forbidden arms, and the STORE_CAP arm of C W-X are model-silent.
5. NOT the overflow regime. Faithful extraction keeps Peano nat, which cannot express uint64 wrap, so the near-2^64 corners are witnessed as divergence-by-design and deferred to CBMC, not adjudicated by the oracle.
6. NOT off-slice equality. re_mint clamps and intersects, the C rejects, so equality holds only on the OK slice, off it only out_cap.valid = 0 is checkable.
7. NOT a closed trusted base. The Coq extractor, OCaml compiler and runtime, and the unproven hand adapter remain trusted. CakeML would close the first, is not used.
8. NOT closure of the caveat. The bridge NARROWS the caveat (it gives the mint path its first check against the exact proven term) but does not close it. The statement the proof is about the abstract model, not the running C, survives intact.
